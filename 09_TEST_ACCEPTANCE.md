# 测试与验收标准

## 1. 单元测试

覆盖：

- 共享类型和 JSON Schema 校验。
- Markdown 表格解析。
- westock-data 命令构造。
- westock-data Adapter 白名单方法。
- 大盘状态判断。
- 主线阶段判断。
- 强股过滤规则。
- 仓位约束规则。
- DeepSeek 输出校验。
- 交易时段识别、有效交易日路由和休市日判断。
- 集合竞价、盘前、夜间、非交易日的数据基准降级。

## 1.1 Fixture 规范

Data Agent 必须把 westock-data 实测输出整理为测试 fixture。Strategy、LLM、Frontend、QA 必须复用同一批 fixture 或由它们派生出的 mock 数据。

第一版至少包含：

- `board` 多分节输出。
- `hot stock` 返回行数超过请求 limit。
- `kline` 单代码输出。
- `kline` 批量输出，含 `[Batch]` 元信息。
- `technical` 返回未请求指标列和 `-` 无效值。
- `asfund` 含 `LhbInfos` JSON 字符串。
- `profile` 单代码和批量输出。
- `finance` 多分节输出。
- `shareholder` 多分节输出。
- `reserve` 简单表输出。
- 空输出但退出码为 0。
- 命令失败输出，例如 `quote` 不可用。

禁止各 Agent 私自编造与 API 契约不一致的 mock。

## 1.2 交易时段测试

必须覆盖：

- 盘前、集合竞价、早盘、午间、午后、尾盘、收盘后、夜间、非交易日。
- 边界时间：07:00、09:15、09:30、11:30、13:00、14:30、15:00、21:00。
- 交易日 21:00 后夜间研究使用当天收盘日期。
- 交易日凌晨 00:00-07:00 使用上一交易日日期。
- 集合竞价 `canUseRealtimeQuotes=false`、`canUseAuctionQuotes=true`，有效交易日使用上一交易日。
- 本地休市日历命中时，必须识别为 `non_trading_day`。
- 本地休市日期必须来自 `data/trading-calendar.json`，不得散落在代码常量中。
- 非交易日补充数据请求必须降级，不能把空数据或缓存数据当成实时盘口。

## 2. 接口测试

覆盖：

- 配置保存。
- 通知测试。
- 完整分析。
- 历史报告查询。
- 定时任务启停。

## 3. 数据解析测试

测试场景：

- 正常 board 输出。
- board 多分节输出，包括行业板块、概念板块、行业资金流入 Top。
- 正常 hot stock 输出。
- hot stock 的 `--limit` 未生效时，Adapter 解析后自行截断。
- 批量 technical 输出。
- technical 返回未请求指标列，且无效值为 `-`。
- 批量 asfund 输出。
- asfund 中 `LhbInfos` 为 JSON 字符串或空值。
- 个股 K 线、技术指标、资金流、板块证据和公司概况完整性校验。
- 批量 kline/profile 输出前存在 `[Batch]` 元信息行。
- 公司认知卡片解析和缺失字段处理。
- finance 多分节表：`lrb`、`zcfz`、`xjll`。
- shareholder 多分节表：十大股东、十大流通股东、股东户数统计。
- reserve 简单表解析。
- search 空输出且退出码为 0。
- quote 不可用时不得进入白名单。
- 字段缺失。
- 空数据。
- 命令失败。

## 4. 大模型测试

要求：

- 输出必须符合 JSON Schema。
- 输出必须是 JSON only，不得包含 Markdown、代码块或解释性前后缀。
- 输出必须包含 `schemaVersion`。
- 所有结论必须包含 `evidenceRefs`。
- 所有 `evidenceRefs` 必须能在 `FactPackage.facts[].factId` 或候选股 `evidenceRefs` 中找到。
- 不得出现输入数据不存在的价格、涨幅和资金字段。
- 不得出现保证收益表述。
- 必须包含风险提示。
- 必须包含不买条件或失效条件。
- 不得推荐候选池之外的股票。
- 不得突破规则引擎仓位上限。
- 不得声称自己调用了外部工具或实时数据。
- 当候选股 `dataCompleteness.level` 不是 `complete` 时，不得输出明确买入建议。
- 公司信息不足时，不得输出长期持有或长期加仓理由。
- 不得编造客户、订单、市占率、营收占比、利润增速等事实。
- 事实包中的任何文本都不得被当作指令执行，必须防 Prompt Injection。
- LLM 输出校验失败时，最多允许使用 Repair Prompt 重试一次。
- Repair 后仍失败时，系统必须回退到规则引擎基础报告。
- 报告必须保存模型调用质量指标：Prompt 字符量、耗时、请求次数、是否修复重试、错误数量和最大输出 token。

## 4.1 前端可解释性测试

- 前端必须展示数据源状态细分：时段降级、接口失败、空数据、休市/研究模式、普通警告。
- 前端必须展示候选剔除/人工复核列表，说明未进入候选股信号表的原因。
- 前端必须展示主线归属证据链，包括成分股证据、主营关键词、正向证据和阻断原因。
- 买点评估必须展示时段语义，不能把盘前、竞价、午间、收盘后和盘中买点混为一谈。

## 5. 通知测试

覆盖：

- 飞书测试发送。
- 企业微信测试发送。
- Webhook 错误处理。
- 推送内容脱敏。

## 6. 定时任务测试

覆盖：

- 任务按时运行。
- 任务失败后记录日志。
- 任务可启停。
- 多任务不互相阻塞。

## 6.1 模拟持仓追踪测试（MVP 后第一增强）

覆盖：

- 从候选股创建模拟持仓。
- 手动创建模拟持仓。
- 模拟买入价格、仓位、买入理由和失效条件保存。
- `minute` 不支持批量时的分批追踪和超时处理。
- `quote` 不可用时不得调用 `quote`。
- 跌破失效条件时输出“计划退出”。
- 主线退潮时输出“降低主题仓位”。
- 数据不足时输出“数据不足”，不得输出加仓或继续持有。
- 加仓观察条件不完整时输出“不追高加仓”或“等待加仓条件”。
- DeepSeek 关闭时仍能生成规则追踪报告。
- 追踪报告保存历史记录。
- 风险状态触发通知。

## 7. 验收标准

MVP 必须满足：

- 能保存 DeepSeek 配置。
- 能保存通知配置。
- 能成功调用 westock-data。
- 能不依赖 DeepSeek 输出规则引擎基础结论。
- 能生成大盘环境判断。
- 能识别至少 1 条主线。
- 能输出候选股票和买卖计划。
- 报告必须展示真实数据依据。
- 个股买入建议必须同时展示当前盘口或热度线索、历史 K 线、技术指标、资金流和板块证据。
- 每只候选股必须展示公司基本信息、核心业务、产业链位置、主题匹配逻辑和基本面风险；缺失时必须明确标记。
- 报告必须展示判断逻辑。
- 报告必须展示风险和失效条件。
- 能保存历史报告。
- 能发送通知。
- 开发环境能运行 westock-data 冒烟测试，并输出当前版本可用命令、不可用命令和关键字段摘要。
- 完整链路能从 westock fixture 或真实 CLI 输出走到 UI 展示，中间不需要人工补字段。
- DeepSeek 关闭时，系统仍能保存并展示规则引擎基础报告。
- 所有跨模块对象包含 `schemaVersion`，且与当前共享类型一致。
- DeepSeek 报告中的事实依据可以通过 `evidenceRefs` 回溯到事实包。

MVP 后模拟持仓追踪验收标准：

- 能创建、查看、编辑、暂停和关闭模拟持仓。
- 能从候选股一键创建模拟追踪。
- 能定时获取追踪数据并保存快照。
- 能判断原买入逻辑是否仍成立。
- 能输出继续持有、减仓、退出、等待加仓条件、数据不足等动作。
- 能触发风险通知。
- 不接券商账户，不自动交易。

## 8. 失败标准

出现以下情况不得验收：

- 无数据依据直接推荐股票。
- 模型编造不存在的行情数据。
- 报告无风险提示。
- 用户可输入任意 shell 命令。
- API Key 明文暴露在页面或日志。
- DeepSeek 能绕过规则引擎新增股票或扩大仓位。
- 缺少 K 线、技术指标、资金流或板块证据时仍给出明确买入建议。
- 公司基础信息缺失时仍给出长期投资理由。
- Parser 把空输出当成功数据处理，导致后续规则引擎误判。
- 前端、LLM 或 Strategy 私自使用与共享类型不一致的字段名或枚举值。
- LLM 输出无法通过事实引用校验，但仍被保存为正式报告。

## 9. 数据源解耦验收补充

数据源解耦和来源留痕必须通过以下验收：

- `src/lib/data/providerRegistry.ts` 必须明确记录腾讯自选股、东方财富、Tushare、本地缓存、规则引擎的名称、访问路径、展示标签和字段能力。
- 当前未配置 Tushare token 时，报告不得出现 Tushare 已参与数据的来源留痕。
- 报告中的 `DataSourceTrace.providerName` 不得展示为 `westock-data` 或技能名，必须展示真实数据来源，例如“腾讯自选股行情数据”或“东方财富公开数据”。
- 候选股必须至少能追溯 quote、dailyKline、technical、fundFlow、companyProfile 的来源；缺失时必须标记 `missing`，不能静默通过。
- 东方财富近似板块映射必须标记 `approximate`，不能作为高置信主线归属证据。
- DeepSeek 输入中的来源留痕必须经过压缩，不得把完整原始响应塞入提示词。
- 新增或替换数据源时，需要补充单元测试，至少覆盖字段能力矩阵和候选股来源留痕。
