# 策略逻辑说明书

## 1. 总原则

系统策略为：

> A 股主线趋势 + 中军优先 + 分歧低吸策略

核心策略判断必须固化在程序中。DeepSeek 不能作为第一判断源，只能基于程序生成的事实包和规则结论进行解释、润色、补充风险提示。

顺序固定：

1. 看市场。
2. 看板块。
3. 看个股。
4. 看买点。
5. 看仓位和风险。

任何推荐必须先给事实依据，再给判断逻辑，最后给建议。

个股建议不能只基于行情指标。每只候选股必须同时输出公司认知摘要，说明公司核心业务、产业链位置、与当前主线的匹配关系和基本面风险。对于中线、长线建议，公司认知缺失时不得给出长期持有理由。

## 1.1 程序化判断边界

必须由程序完成：

- westock-data 命令调用。
- 数据解析。
- 大盘环境初判。
- 主线板块候选生成。
- 强股候选过滤。
- 买点类型识别。
- 硬性风控和仓位上限约束。

可以由 DeepSeek 完成：

- 把结构化结论解释成人话。
- 比较多个候选的优先级。
- 输出不同交易风格的建议文本。
- 补充风险提示。
- 生成通知摘要。

DeepSeek 不得完成：

- 自行调用 westock-data。
- 自行发明股票池。
- 自行编造资金、价格、涨跌幅、均线、新闻。
- 绕过程序风控给出更激进仓位。

## 2. 大盘环境判断

### 2.0 交易时段上下文

规则引擎必须先识别当前 A 股交易时段，再解释数据和生成报告。系统不得全天使用同一套“盘中盯盘”口吻。

`MarketSessionContext` 至少包含：

- `phase`：`premarket | call_auction | morning | midday_break | afternoon | closing_auction | postmarket | night_research | non_trading_day`
- `phaseLabel`：盘前计划 / 集合竞价 / 早盘盯盘 / 午间复盘 / 午后确认 / 尾盘确认 / 收盘复盘 / 夜间研究 / 非交易日研究
- `analysisMode`：计划 / 竞价观察 / 盘中盯盘 / 半日复盘 / 尾盘决策 / 收盘复盘 / 深度研究
- `expectedDataBasis`：上一交易日收盘、竞价数据、盘中实时/延迟行情、上午收盘快照、尾盘行情、当日收盘数据或历史数据
- `isTradingDay`、`isTradingSession`、`isIntraday`
- `canUseRealtimeQuotes`、`canUseAuctionQuotes`
- `ruleFocus`、`llmFocus`、`outputRestrictions`

时段规则：

- 盘前：只能基于上一交易日收盘结构生成今日验证清单，不得声称今日盘口已确认。
- 集合竞价：只作为弱确认，不能替代开盘后承接；`canUseRealtimeQuotes=false`，`canUseAuctionQuotes=true`，涨跌停池优先使用上一交易日数据。
- 早盘：适合发现主线启动，但不能把早盘单点强势写成收盘确认。
- 午间：必须总结上午验证情况，输出下午翻转条件和失效条件。
- 午后：重点确认早盘线索是否延续、资金是否回流、核心股是否修复。
- 尾盘：重视收盘位置和隔日风险，不得追尾盘无承接脉冲。
- 收盘后：写正式复盘和次日验证条件，盘中盯盘任务应转为明日观察项。
- 夜间/非交易日：只做历史复盘、公司认知、候选池维护和策略研究，不做实时盘口判断；交易日 21:00 后夜间研究使用当日收盘日期，凌晨 00:00-07:00 使用上一交易日日期。

交易日历和数据日期：

- `isTradingDay` 不能只按周一到周五判断，必须接入 A 股休市日历；第一版可使用本地维护的休市日期表，后续可替换为交易所日历数据源。
- 本地休市日期应维护在 `data/trading-calendar.json`，代码不得再散落硬编码休市日。
- `effectiveTradeDate` 必须按时段生成：盘前、集合竞价、非交易日使用上一交易日；交易日夜间 21:00 后使用当天；交易日凌晨使用上一交易日。
- 非交易日不得把空数据或缓存数据当作实时行情；补充数据请求应降级，并在事实包中写入数据基准警告。

DeepSeek 可以使用 `FactPackage.session` 调整报告结构和措辞，但不得改变规则引擎的大盘状态、主线阶段、候选池、买点状态、仓位上限和风控约束。

### 2.0.1 大盘状态原因

`marketState=defensive` 必须同时输出 `marketStateReason`，避免把数据故障误读成真实弱势：

- `数据不足防守`：指数、宽度或关键情绪数据不足，系统按防守处理，但不能断言盘面真实走弱。
- `风险事件防守`：跌停/大跌风险显著、主线退潮等风险项触发。
- `真实弱势`：指数、宽度、情绪和主线结构共同指向弱势。
- `正常评估`：非防守状态，按完整评分正常判断。

### 2.1 输入

- 上证指数：`sh000001`
- 深成指：`sz399001`
- 创业板指：`sz399006`
- 科创 50：`sh000688`

### 2.2 指标

- 分时是否高位横住。
- 是否冲高回落。
- 是否持续走低。
- 日线是否在 20 日线、60 日线之上。指数 MA 优先使用 westock `technical` 返回的 `ma.MA_20`、`ma.MA_60`；K 线自算均线只作为 fallback。
- 指数 K 线至少获取 90 条，禁止用 30 条 K 线计算并命名为 MA60。
- 指数 K 线必须按日期倒序标准化后再计算最近 N 日指标，不能依赖数据源返回顺序。
- 均线排列状态：多头排列、空头排列、非单边排列。
- MA20 斜率、20 日动量、20 日量能比和 20 日波动率。
- 成交金额是否放大或萎缩，特别关注上涨缩量、下跌放量。
- 成长指数是否强于主板。
- 核心指数共振程度，区分主板指数（上证、深成指）与成长指数（创业板、科创50）的健康分化和危险分化。
- 全 A 市场宽度是大盘状态的强证据；板块涨跌比和热股正收益比例只能作为弱 fallback，不能替代全 A 宽度。
- 宽度证据必须输出可靠性：`market=1.0`，`sector=0.45`，`hot=0.25`，`none=0`。
- 全 A 宽度缺失时，即使板块或热股 fallback 显示偏强，也不得直接判定为可交易环境。
- 板块 fallback 必须按成分股数量加权，禁止小板块与大板块简单平均。
- 热股 fallback 需要最小样本数，样本过少时视为无可靠宽度证据。
- 大涨/大跌家数阈值应随全 A 总数动态调整，避免市场扩容后固定阈值失真。
- 大盘状态必须加入“宽度连续性”记忆：每次报告保留全 A 上涨占比、中位涨跌幅、宽度评分、证据来源和可靠性；最近报告之间上涨占比改善或恶化超过 8 个百分点时，标记为“宽度改善/宽度转弱”。该连续性只能作为边际变化证据，不能替代今日全 A 宽度、指数结构和风控硬约束。
- DeepSeek 使用 `memory.market.breadth_timeline` 时，只能解释“赚钱效应是否边际改善/恶化”和“大盘状态是否有翻转迹象”，不得因为历史宽度改善而突破当日规则仓位。
- 涨跌停情绪必须区分数据来源：`pool` 表示真实涨停/跌停/炸板池，`approx` 表示全 A 宽度中的近似涨停/跌停，`missing` 表示缺失。近似数据只能作为弱参考，缺失数据不得按 0 风险处理。
- 当涨停池、跌停池、炸板池和全 A 大跌数据全部缺失时，情绪分必须为 0，诊断标记为缺失，不得因 `dt=0/bigDown=0` 给出安全加分。
- 炸板池缺失时，炸板风险不能视为 0，只能降低情绪评分可靠性；若炸板池可用，炸板率应按分档扣分，而不是只用单一阈值。
- 跌停数与大跌家数应独立分档叠加，避免 `dt` 或 `bigDown` 其中一个接近危险区但未被扣分。`bigDown >= 400` 与 `dt >= 40` 统一视为恐慌风险，强制大盘进入防守。
- 情绪诊断必须输出涨停、跌停、炸板、大跌数量及其来源和可靠性，用户必须能看出当前情绪数据是精确池、近似数据还是缺失。

### 2.3 结论分级

#### 可交易

满足多数条件：

- 指数不破早盘关键低点。
- 深成指、创业板或科创方向有承接。
- 主线板块资金净流入明显。
- 板块涨跌扩散，不是单一股票独涨。

程序输出字段建议：

```json
{
  "marketState": "tradable",
  "score": 78,
  "indexResonance": 0.86,
  "facts": [],
  "riskFlags": []
}
```

#### 谨慎交易

出现：

- 指数冲高回落。
- 主线仍在，但高位股分歧。
- 成交不够，资金只集中少数票。

程序输出字段建议：

```json
{
  "marketState": "cautious",
  "score": 50,
  "facts": [],
  "riskFlags": []
}
```

#### 防守观望

出现：

- 指数跌破关键均线或分时单边下跌。
- 主线板块资金流出。
- 强股补跌。
- 亏钱效应扩散。

程序输出字段建议：

```json
{
  "marketState": "defensive",
  "score": 20,
  "facts": [],
  "riskFlags": []
}
```

## 3. 主线识别规则

主线不是单日涨幅第一。主线必须同时具备：

- 板块涨幅靠前。
- 主力资金流入靠前。
- 5 日或 20 日涨幅有持续性。
- 板块内上涨家数较多。
- 有龙头、有中军、有补涨。
- 分歧后能修复。

### 3.1 阶段判断

主线阶段不能每天孤立判断。规则引擎先根据当日价格、资金、成分扩散、涨停核心和排名计算 `rawStage`，再结合历史时间链的同名主线阶段做迁移校验，输出最终 `stage`。

历史连续性窗口：

- 短线窗口：最近 3 次历史报告 + 本次报告，用于判断大盘状态、宽度和主线阶段的边际改善/恶化。
- 中线窗口：最近约 10 次历史报告 + 本次报告，用于判断主线持续性、核心股是否频繁更换、同名主线是否反复启动失败。
- 质量窗口：最近约 20 份已保存报告，用于统计可展示报告、过滤报告、解析失败和时间跨度，输出 `memory.market.timeline_quality`。
- 如果时间链可靠性为中/低，规则和 DeepSeek 都只能降低连续性结论置信度，不能把数据断档误判为主线自然退潮，也不能因为历史点不足而强行得出“持续改善”。
- DeepSeek 只接收压缩后的时间链、阶段路径和质量摘要，不接收全部历史原文，避免 token 随时间无限膨胀。

阶段迁移约束：

- `观察/启动 -> 确认`：必须同时满足较高综合评分、资金强度、成分扩散和涨停核心证据；否则先压制为 `启动`，等待下一期验证。
- `观察/启动 -> 加速`：不能一日跳级为加速；即使当日很强，也应先修正为 `确认` 或 `启动`，防止把单日高潮误判为成熟主线。
- `确认/加速 -> 观察`：若综合评分仍有一定支撑，先标记为 `分歧`，观察核心股、资金和扩散是否修复；只有资金、扩散、核心结构同步恶化时才降为观察或退潮。
- `退潮` 不做乐观修正。出现核心股补跌、资金持续流出、扩散坍塌时，必须优先保护风险。

核心股持续性约束：

- 每条主线必须记录本期核心股与上一期同名主线核心股的关系：延续、新出现、退出、上一期龙头、本期龙头、是否换龙头。
- 龙头强但中军和延续核心缺失，只能说明短线情绪强，不能直接说明主线确认。
- 核心股至少有 1-2 只延续，且本期有新核心补充，视为“轮动健康”；核心股连续保留且龙头未切换，视为“稳定”。
- 若上一期为确认/加速，本期仍被当日规则判为确认，但核心股完全更换且龙头切换，最终阶段应先修正为 `分歧`，等待新核心连续性验证。
- 若核心股频繁更换、旧核心退出、新核心没有连续涨停或中军承接，应降低主线可信度，不能生成“主线持续确认”的强结论。

板块命名与同义主线映射：

- 板块进入规则引擎前必须通过 `src/lib/sector/normalization.ts` 生成 `normalizedName`，并保留 `sourceNames`。如果多个来源名被合并，报告必须能展示原始来源名。
- 规则引擎、历史报告/记忆、东方财富板块匹配必须共用同一套归一化模块，禁止在各模块内复制一份局部别名表，避免同一板块在规则和记忆里断裂。
- 只合并高度确定的同义或强等价名称，例如：`元件/被动元件/电子元件/电子元器件`、`电子特气/电子气体/特种气体/工业气体`、`焦炭/焦炭Ⅱ`、`煤炭/煤炭开采`、`通信设备/通讯设备/网络设备`、`空心杯电机/机器人执行器`、`光刻机/半导体设备`、`光刻胶/半导体材料`。
- 不得把“产业链相关”直接等同于“同一主线”。例如 PCB、半导体、国家大基金持股、消费电子、元件之间可能存在资金联动，但不能默认合并；只能在 DeepSeek 的主线竞争关系中解释为科技链扩散或资金轮动。
- 名称归一化只解决“同一板块不同写法”；产业链上下游、风格扩散、资金竞争关系，必须交给主线归属证据链和 DeepSeek 结构解读，不能通过 `normalizedName` 强行合并。
- 外部数据源若无法找到原始板块而使用别名或近似板块取得成分股，必须输出 `approximate` 或 warning，并在个股主线归属中降级确认，不能当作直接成分证据。
- 候选股主线归属、成分股匹配、历史阶段迁移和核心股延续，都必须使用同一套 normalizedName，否则会出现今天叫 A、明天叫 B 导致记忆断裂。

输出字段建议：

```json
{
  "rawStage": "确认",
  "previousStage": "启动",
  "stage": "启动",
  "stageTransition": "压制升级",
  "stageTransitionReason": "上一期为启动，本期原始阶段直接升为确认，但资金、扩散、涨停核心未同时达到强确认阈值，先按启动处理。",
  "coreContinuity": {
    "retained": ["中军A"],
    "appeared": ["新龙头B"],
    "disappeared": ["旧龙头C"],
    "leaderChanged": true,
    "state": "换龙头待确认",
    "reason": "上一期核心为旧龙头C、中军A；本期核心为新龙头B、中军A；龙头切换但中军延续，先观察新核心能否连续。"
  }
}
```

#### 启动

- 板块刚放量。
- 5 日涨幅不高。
- 龙头开始突破。

#### 确认

- 板块连续多日靠前。
- 龙头、中军、补涨同时出现。
- 主力资金持续流入。

#### 加速

- 龙头连续大涨或涨停。
- 板块涨幅很高。
- 后排开始补涨。
- 风险上升，不适合追涨。

#### 分歧

- 龙头炸板或冲高回落。
- 后排掉队。
- 中军仍有承接。

#### 退潮

- 龙头补跌。
- 板块资金流出。
- 前期强股跌破 10 日或 20 日线。

## 4. 强股筛选规则

### 4.1 必须条件

- 处于当前主线或主线分支。
- 已获取历史 K 线、技术指标、资金流、板块证据和公司基础信息；只有当前盘口数据时，不得给出买入建议。
- 公司主营业务与当前主线存在可解释的匹配关系，或明确标记为“仅交易性题材，不具备长期逻辑确认”。
- 股价在 20 日线之上，或刚重新站回 20 日线。
- 不能处于明显下降趋势。
- 当日或近 5 日主力资金有流入迹象。
- 流动性足够。

### 4.1.1 主线归属证据链

候选股不能因为热门股排行靠前、涨幅靠前，或系统当前第一主线较强，就被自动归入该主线。规则引擎必须先输出 `mainlineAttribution`，再决定是否进入候选股信号表。

归属等级：

- `direct_constituent`：数据源证明个股属于当前主线行业/概念成分股，置信度高，允许进入候选池。
- `business_direct`：缺少成分股证据，但公司主营、行业或业务描述与当前主线关键词直接匹配，置信度中，允许进入候选池但必须提示“缺少成分股直接证据”。
- `supply_chain_related`：仅产业链上下游、材料、设备、客户场景或题材弱相关，置信度低，不得自动进入候选池，只能作为人工复核线索。
- `theme_indirect`：题材间接相关但无主营或成分股证据，不得进入候选池。
- `mismatch`：主营业务与主线明显偏离，必须剔除。
- `unknown`：公司基础信息或板块证据缺失，必须剔除。

程序必须保存并展示：

- `matchedSector`：尝试匹配的主线。
- `membershipSector`：数据源返回的成分板块名称。
- `businessKeywords`：从主营/行业中命中的关键词。
- `sectorKeywords`：该主线允许匹配的关键词集合。
- `evidence`：纳入证据。
- `blockers`：阻断原因。
- `evidenceChain`：结构化证据链，必须包含：
  - `constituentEvidence`：成分股/板块归属证据。
  - `businessEvidence`：主营业务、行业、产品关键词证据。
  - `industryChainEvidence`：产业链位置或上下游弱相关证据。
  - `negativeEvidence`：否定证据，例如主营偏离、缺少成分股证据、仅弱相关。
  - `sourceQuality`：`direct|inferred|weak|missing`。
  - `reviewRequired`：是否需要人工复核。
  - `reviewReason`：复核原因。
- `confidence`：高/中/低。
- `shouldExclude`：是否剔除。
- `reason`：给用户看的中文结论。

被剔除或待复核的股票不得直接丢弃。系统必须在 `FactPackage.candidateReviews` 中保留结构化记录：

- `code/name`：股票代码和名称。
- `status`：剔除或人工复核。
- `reason`：未进入信号表的原因。
- `missingEvidence`：缺少哪些证据，例如主线成分股证据、公司基础信息、资金流、K 线或技术指标。
- `blockers`：阻断条件。
- `evidence`：已有证据。
- `evidenceChain`：同 `mainlineAttribution.evidenceChain`。
- `attributionStatus`：归属状态。
- `reviewRequired`：是否需要人工复核。

重要约束：

- `chooseCandidateSector` 不允许在无证据时回退到第一主线。没有成分股、主营直接匹配或明确证据时，应返回待确认并剔除。
- 即使存在成分股证据，如果公司主营资料没有直接命中主线关键词，也应在 `negativeEvidence` 中记录“主营未直接命中”，并把 `reviewRequired` 标记为 true，但不直接误杀成分股。
- “产业链相关”不是“主线核心”。例如显示面板、PCB、电子材料等可以作为元件链条线索，但若没有元件成分股证据或主营直接命中元件关键词，不得自动归入元件主线。
- DeepSeek 只能解释 `mainlineAttribution`，不能自行改变归属等级、推荐候选池外股票，或把 `shouldExclude=true` 的股票写成买入计划。

### 4.2 优先级

1. 主线中军。
2. 主线趋势核心。
3. 分歧后率先修复的强股。
4. 低位补涨但资金确认的股票。

个股定位不能只依赖候选数组顺序。程序应按以下顺序推断 `role`：

1. 优先匹配主线规则输出的 `coreStocks`，若命中则使用核心股角色：龙头/中军/补涨。
2. 若未命中核心股列表，但来自当前主线成分股前排，则按板块内涨幅和成交额排名进行保守定位：前排可为中军或补涨，不能轻易标记龙头。
3. 若只是热门股且依赖主营业务匹配主线，默认只能是低吸观察，不能因为在热门股列表靠前就标记龙头。
4. 每只候选股必须输出 `rule.stock.{code}.role` 事实，说明定位依据，供 DeepSeek 和前端引用。

### 4.2.1 候选池排序与信号质量分层

候选股信号表不得直接沿用热股榜、数据源返回顺序或人工录入顺序。候选池需要先完成主线归属、强股评分、买点评估、动作判定和风控约束，再按 `signalScore` 做统一排序。

排序输入包括：

- 规则动作：`小仓试错` 优先于 `等待回踩`、`观察`、`不追`、`回避`、`数据不足`。
- 强股评分：趋势、资金、主线、角色、买点、风险扣分后的综合强度。
- 买点质量：有效买点、待激活买点、缺证据买点、无效买点必须分层。
- 数据完整性：核心数据完整优先，缺少 K 线、技术指标、资金流或板块证据必须降级。
- 主线归属：直接成分股和主营直接匹配优先；产业链弱相关、主题间接、错配必须降权或剔除。
- 活跃度：不强依赖量比字段。优先使用现成的成交额、换手率、主力净流/资金质量、板块内成交额或涨幅前排、涨停/炸板状态、均线距离来判断资金参与和筹码交换质量。
- 买入可达性：强股不等于能买。若个股已经接近涨停或涨停不可达，即使属于主线核心，也只能标记为 `不追/观察`，仓位为 0，等待开板换手回封、分歧回踩或次日承接确认。
- 个股角色：龙头/中军优先于补涨，低吸观察只能作为条件候选。
- 大盘和主线阶段：可交易/谨慎、启动/确认/加速可加分；防守、分歧、退潮必须压制。
- 风险标记：趋势破位、资金流出、远离均线、主题偏离、数据不足等必须扣分。

信号质量分层：

- `S / 核心试错`：规则动作明确为 `小仓试错`，买点有效，核心数据不缺失，主线归属不被剔除，排序分达到高位。它仍然必须服从 `positionLimitPct`。
- `A / 重点观察`：结构质量较高、风险不硬伤，但尚未满足完整小仓试错条件，可作为盯盘重点。
- `B / 条件等待`：趋势或主线结构存在，但需要大盘、主线、买点或资金触发条件进一步确认。
- `C / 风险压制`：不追、防守压制、风险较多或买点缺失，只能解释为什么暂不参与。
- `D / 剔除/低质`：数据不足、回避、主线错配或证据弱，不应占用核心候选展示位置。

硬约束：

- `signalTier` 只用于候选池排序、前端展示和解释优先级，不能改变 `action`、`positionLimitPct`、`riskFlags`、`buyPointEvaluation` 或主线归属。
- `activity` 只代表交易活跃和承接证据，不等同于买点。活跃度强但涨停不可达、远离均线、资金流出或大盘防守时，仍必须按风控输出 `不追/观察/回避`。
- `观察`、`不追`、`回避`、`数据不足` 即使排序靠前，仓位仍必须为 0%。
- 每只候选股必须输出 `rule.stock.{code}.activity` 和 `rule.stock.{code}.signal_quality` 事实，说明活跃度、分层、排序分和主要依据，供 DeepSeek、反馈模块和前端详情页引用。

买入可达性规则：

- `涨停不可达`：涨跌幅接近或达到 10% 涨停区间，普通实盘大概率买不进去，动作必须为 `不追`，仓位 0。
- `接近涨停`：涨幅接近涨停区间，追价成交质量差，动作必须为 `不追`，仓位 0。
- `高位拉升`：日内涨幅偏高，不能用追涨价作为买点，只能等待回踩、分歧修复或尾盘承接确认。
- `可买入观察`：仍需同时满足主线、买点、资金、风控和仓位约束，不能单独触发买入。

涨停票不是永久排除。若个股属于主线核心且今日涨停不可达，系统应生成 `nextSessionPlan`：

- `mode=次日竞价观察`：只代表明天需要验证，不代表今天或明天无条件买入。
- 前提条件必须包括：次日不是一字买不到、不是明显低开弱转强失败、竞价成交活跃但不过度高开、所属主线继续启动/确认、核心股没有大面积负反馈、开盘后承接不破分时均价线或回踩关键均线后修复。
- 不追条件必须包括：一字板排队追、高开过多后快速下杀、板块核心股炸板、后排补跌、大盘转防守。
- 失效条件必须包括：竞价低于预期且开盘不能收回、开盘放量下杀跌破分时均价线后无修复、主线退潮或资金明显流出。
- 在次日竞价和开盘数据真正出现前，DeepSeek 只能输出“次日观察计划”，不能把它写成明确买入建议。

### 4.3 剔除条件

- ST、*ST。
- 跌破 20 日线且 20 日线下行。
- 跌破 60 日线且无资金修复。
- 连续大涨远离 5 日线，处于高潮阶段。
- 资金连续 5 日和 10 日流出。
- 龙虎榜显示强分歧但没有承接。
- 缺少历史 K 线、技术指标、资金流或板块证据，导致无法验证趋势和主线位置。
- 公司基础信息缺失，或主营业务与所处热点明显不匹配且无其他事实依据。

## 5. 买点规则

买点识别必须分两层：

1. **结构识别**：先独立计算个股是否出现回踩均线、突破回踩、分歧修复等潜在结构。
2. **风控激活**：再由大盘状态、主线阶段、资金流、远离均线程度和仓位规则决定是否允许执行。

因此，防守行情下不能简单丢弃买点结构，而应标记为“待激活”；但仓位仍然必须为 0%，不得因为存在潜在买点而突破风控。

程序字段建议：

```json
{
  "buyPointType": "回踩均线",
  "buyPointEvaluation": {
    "type": "回踩均线",
    "score": 14,
    "status": "有效|待激活|无效|缺证据",
    "satisfied": ["趋势站上MA20", "贴近MA10", "资金未连续流出"],
    "blockers": ["大盘防守，买点只记录为待激活"],
    "triggerCondition": "大盘至少修复至谨慎交易，股价贴近MA10/MA20不破且主力资金不连续流出。",
    "invalidCondition": "跌破MA20、资金连续流出或主线退潮，则回踩买点失效。"
  }
}
```

买点状态解释：

- `有效`：结构满足，主线阶段允许，大盘不是防守，且无资金连续流出、趋势破位或远离均线等硬阻断。
- `待激活`：结构存在，但大盘防守、主线阶段暂不允许或仍需确认，只能观察和设置条件，不给仓位。
- `无效`：无明确结构，或出现趋势破位、资金连续流出、主线退潮、远离均线追高等硬阻断。
- `缺证据`：缺少 K 线、技术指标或资金流，不能识别买点。

### 5.0.1 买点时段闸门

买点识别必须叠加 `FactPackage.session.phase`，每个买点输出 `sessionNote`：

- 盘前：只生成今日验证清单，不确认今日买点。
- 集合竞价：竞价异动只作弱参考，必须等待 9:30 后承接。
- 早盘：可识别启动线索，但需要午后和尾盘继续验证。
- 午间：只形成下午验证条件，不声称下午已经确认。
- 午后：重点验证早盘线索是否延续、资金是否回流。
- 尾盘：必须额外提示收盘位置、隔日低开和尾盘脉冲风险。
- 收盘后/夜间/非交易日：只输出次日或下个交易日验证条件，不输出实时买点确认。

### 5.1 A 类买点：回踩均线

- 主线仍强。
- 个股回踩 5 日或 10 日线不破。
- 成交缩量或有承接。
- 主力资金没有明显流出。
- 若大盘防守，只能记录为“待激活”，触发条件为大盘至少修复至谨慎交易。

买点优先级以风险收益比为先：

1. 回踩 MA20 或关键均线不破，优先级最高。
2. 突破后回踩 MA10/突破位不破，优先级次之。
3. 板块分歧时个股仍站上 MA20，可标记为分歧修复。
4. 个股刚收复 MA20 风险最高，只能低优先级观察，不能优先于明确回踩买点。

远离均线阈值必须区分两类语义：

- 趋势位置阈值：MA5 距离超过 10% 或 MA20 距离超过 18%，趋势评分降级。
- 买点追高阈值：MA5 距离超过 8% 或 MA20 距离超过 14%，买点降级为无效或待确认。
- 两套阈值可以不同，但必须在诊断中说明，不能让同一只股票同时表现为“买点无效但无追高风险”。

### 5.2 B 类买点：突破回踩

- 放量突破平台或前高。
- 回踩突破位不破。
- 板块未退潮。
- 如果股价远离 MA5 或 MA20，即使强势也标记为“不追/无效”，等待下一次回踩。

### 5.3 C 类买点：分歧修复

- 板块早盘分歧。
- 中军率先修复。
- 龙头没有集体补跌。
- 适用于主线分歧或个股刚收复 MA20 的场景，要求资金不能连续流出。

### 5.4 候选股综合评分口径

候选评分用于排序和解释，不得绕过动作决策和仓位约束。

主线环境评分按“环境质量”而不是“介入风险”计算：

- 加速：20
- 确认：18
- 启动：14
- 分歧：8
- 观察：4
- 退潮/缺失：0

加速阶段环境强但介入风险高，风险必须通过买点评估、allowedBuyTypes、禁止动作和仓位规则处理，不能在环境评分里把加速压得比分歧更低。

资金承接必须使用多周期判断：

- 当日、5 日、10 日、20 日主力资金共同判断 `inflow / mixed / outflow`。
- 当日与 5 日、20 日同向流入，或 20 日流入且短期至少一个窗口流入，才可偏向 `inflow`。
- 当日与 5 日、20 日同向流出，或 20 日流出且当日也流出，才可偏向 `outflow`。
- 当日流入但 10/20 日流出，或短中周期方向不一致，应标记为 `mixed`，不得直接视为强流入。
- 资金评分应在状态基础上加入梯度：20 日资金、10 日资金、超大单/大单资金可加分；中期流出和短期流出共振要扣分。

### 5.5 不买条件

- 涨停附近追入。
- 高位连续大阳后远离均线。
- 板块高潮日后排补涨。
- 大盘分时跳水。
- 没有止损位。
- 只有当前盘口或热门数据，没有历史走势、技术指标、资金流和板块证据支撑。
- 只知道股票涨，不知道公司做什么、靠什么赚钱、为什么匹配当前主线。

## 6. 卖出规则

- 跌破买入逻辑，卖。
- 跌破 10 日线且无法收回，减仓。
- 跌破 20 日线，趋势交易退出。
- 主线退潮，降低主题仓位。
- 高位放量滞涨，减仓。
- 利润大幅回吐时优先锁定部分收益。

## 7. 仓位规则

仓位上限必须由规则引擎给出。DeepSeek 可以解释仓位原因，但不得突破规则引擎给出的上限。

硬约束建议：

- 单只股票建议仓位按 `POSITION_RISK_PROFILE` 控制，默认 `standard`：进攻模式不超过 10%；保守档不超过 8%；进攻档不超过 12%。
- 单一主题总仓位不超过 30%-35%。
- 谨慎行情至少保留 30% 现金。
- 可交易行情至少保留 20% 现金。
- 高位加速票只允许小仓观察。

明确买入动作必须分层：

- 防守状态：不允许开新仓，潜在买点只能标记为待激活。
- 谨慎状态：只允许确认主线的龙头/中军在有效买点、资金不流出、强度达到阈值时给 1%-2% 级别小仓试错。
- 可交易状态：满足有效买点和强股条件时，按当前风险档位给单票上限。

## 8. 多交易风格输出

同一只股票可输出四类建议：

- 短线：1-5 天，重点看盘口、资金、5/10 日线。
- 波段：1-4 周，重点看 20 日线和主线持续性。
- 中线：1-3 个月，重点看行业逻辑和 60 日线。
- 长线：3 个月以上，重点看基本面、景气度、估值和政策。

## 9. 公司认知规则

公司认知卡片用于防止用户只按指标买入。规则引擎必须为每只候选股输出公司认知状态：

```json
{
  "companyKnowledgeState": "sufficient|partial|missing",
  "businessThemeMatch": "strong|medium|weak|unknown",
  "longTermLogicAllowed": true
}
```

判断原则：

- `sufficient`：已获得公司概况，能够说明主营业务、行业和主题匹配关系。
- `partial`：只获得部分基础信息，可以给短线/波段交易计划，但长期逻辑必须谨慎。
- `missing`：公司信息缺失，不得给长期持有或长期加仓理由。

长期或中线建议必须额外说明：

- 公司核心业务。
- 产业链位置。
- 当前主线是否真的受益。
- 需要跟踪的基本面事件。
- 可能证伪长期逻辑的条件。

## 10. 模拟持仓追踪规则（MVP 后第一增强）

模拟持仓追踪用于买后纪律监督，核心判断不是“涨了还是跌了”，而是“原买入逻辑是否仍成立”。

### 10.1 输入

- 模拟买入价格。
- 模拟仓位或数量。
- 买入时间。
- 买入理由。
- 所属主线。
- 买点类型。
- 初始买入条件。
- 初始失效条件。
- 初始减仓条件。
- 初始公司认知卡片。
- 最新分时、K 线、技术指标、资金流、板块和大盘状态。

### 10.2 持续持有条件

可以继续持有或继续观察的条件：

- 大盘未进入防守状态。
- 所属主线未退潮。
- 个股未跌破原计划关键均线或失效位。
- 主力资金没有连续明显流出。
- 买入逻辑仍能被当前数据验证。
- 公司认知没有出现基础信息缺失或主题明显不匹配。

### 10.3 减仓条件

出现以下情况，输出“风险升高，考虑减仓”：

- 大盘由可交易转谨慎。
- 所属主线进入分歧。
- 个股跌破 10 日线且无法收回。
- 高位放量滞涨。
- 主力资金连续流出。
- 模拟盈利大幅回吐并接近原计划防守线。

### 10.4 退出条件

出现以下情况，输出“跌破失效条件，计划退出”：

- 跌破原买入逻辑。
- 跌破 20 日线且趋势交易失效。
- 所属主线退潮。
- 大盘进入防守状态且个股补跌。
- 数据完整性变为 `insufficient`，无法验证继续持有逻辑。

### 10.5 加仓观察条件

第一版不输出激进加仓结论。只有同时满足以下条件，才可输出“满足加仓观察条件”：

- 大盘不是防守状态。
- 所属主线未退潮。
- 原买入逻辑仍成立。
- 个股未高位严重远离 5 日线或 10 日线。
- 资金没有连续明显流出。
- 当前模拟仓位未超过规则仓位上限。
- 出现新的明确买点，例如回踩均线不破、突破回踩确认或分歧修复。

不满足上述条件时，只能输出“不追高加仓”或“等待加仓条件”。

## 阶段迁移：多路径硬确认

主线阶段迁移不能只用单一硬阈值确认。A 股主线经常存在“先龙头涨停，再中军跟随，最后成分扩散”的节奏，因此从 `观察/启动` 升级到 `确认` 时采用多路径确认：

- **广度确认路径**：总分、资金、成分扩散、涨停核心和核心股锚点同时较强，适合成熟主线确认。
- **涨停核心确认路径**：涨停核心明显、龙头或核心股强、炸板压力不大、资金不明显拖累，允许在成分扩散尚未完全成熟时从启动进入确认。
- **连续性确认路径**：历史记忆显示主线已启动，核心股保留度较高，当前仍有涨停核心和资金支撑，允许用连续性补足单日扩散不足。

禁止条件：

- 核心结构处于“换龙头待确认”或“结构偏弱”时不得硬确认。
- 炸板压力明显高于涨停核心时不得走涨停核心确认。
- 资金评分过低且扩散、涨停核心均不足时不得硬确认。
- 该规则只影响主线阶段，不突破大盘风控、个股买点和仓位约束。

## 个股资金流质量

个股资金状态不能只按当日或 5 日资金正负号判断。系统保留 `fundFlowState = inflow | mixed | outflow | unknown` 作为动作判断的简化状态，同时新增 `fundFlowQuality` 用于评分、解释和风险阻断。

资金流质量至少考虑：

- 当日主力净流。
- 5 日、10 日、20 日主力净流。
- 多周期资金方向是否一致。
- 按周期折算后的日均资金强弱，避免“当日小流入掩盖 5 日/10 日大幅流出”。
- 超大单和大单是否与当日主力净流方向一致。

质量状态：

- `强流入`：多周期流入一致，且当日/中期资金没有明显冲突。
- `温和流入`：资金偏正，但强度或连续性不够极致。
- `弱修复`：当日流入但 5 日或 10 日仍明显流出，只能视为修复，不视为健康流入。
- `分歧`：多周期资金方向冲突，或当日流入但超大单/大单存在分歧。
- `持续流出`：当日和中期资金持续流出。
- `未知`：资金流关键字段缺失。

规则影响：

- 强股评分中的“资金承接”优先使用 `fundFlowQuality.score` 映射，而不是只使用 `inflow/mixed/outflow`。
- `弱修复` 和 `分歧` 不得被当作健康流入。
- 资金质量阻断项必须进入候选股风险提示和事实包，供 DeepSeek 研判引用。
